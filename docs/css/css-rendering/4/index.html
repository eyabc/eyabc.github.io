Transform 3D & SCSS & Compass#

CSS 는 언어 차원을 넘어서 프로그램 랭귀지, 컴파일 랭귀지로 바뀌고 있다 CSS 스펙도 점점 프로그래밍이 가능한 상태로 바뀌고 있는중 기존은 변수를 안썼는데, 불편해서 precompiler 가 등장해서 변수를 사용할 수 있게 도와줬다라고 하면 앞으로는 이미 CSS 스펙 자체에 변수가 들어가는 식으로 반영될 것이다.

컴파일러에 대해 이해할 필요가 있다. 어떤 소스를 새로운 소스로 바꿔주는 것을 모두 컴파일이라고 부를 수 있다. target, detination

새로운 언어인 확장 CSS 언어를 작성하면 이것을 CSS 로 바꿔주는 컴파일러들이 있다. 트위터에서 만든 less 오픈소스로 공개되어 있는 SASS (.scss) [super css] css 의 확장셋이다. compass sass 에서 돌아가는 라이브러리 모임이다.

3D 는 소비하지만 생산해 본적은 거의 없을 것이다. 3D 를 구현하려면 2D 보다 더 많은 개념이 필요하다.

단지 2D 에서도 수학적인 공식이 많이 필요하다. 대표적으로 이진 함수를 들 수 있다. CSS 애니매이션을 쓸 때 타이밍 함수라고 이진 함수를 쓰는데, 이것이 2차 베지어 곡선으로 되어있다. 수학함수를 정확하게 알지는 못해도 그런 개념을 사용할 수 있는 정도를 사용할 수 있는것이 목표이다.

Post Process#

그래픽 파이프 라인은 두개를 배웠다. geometry(reflow) 와 fragment(repaint) 이 외에도 post process 라는 것이 있다. geometry 와 fragment 가끝났더라도 바로 칠하지 않고 한번 더 가공할 수 있는 찬스를 준다.

fragment 가 완성되어 있는 픽셀의 데이타라고 생각을 해보자 여기서 얼마든지 후 변형을 줄 수 있다.

리플이라는 수학적 공식으로 픽셀을 흐트려 뜨리고 블러라는 공식을 통해서 완성되어 있는 픽셀값을 변형시킬 수 있다.

픽셀의 색깔, 밝기, 블러, 필터, 그 외에 수많은 변형을 가할 수 있다. 회오리 치게 만들 수도 있고, 물결치게 만들수도 있고 포스터 라이즈를 할 수 도 있고

이런것들을 하기 위해서는 버퍼라는 개념이 필요하다 계산해서 바로 그래픽 비트맵에 쓰는 것이 아니라 메모리에 넣고서 메모리에서 조작한 이후에 그림을 그리는 개념이다

기존의 브라우저는 버퍼의 개념이 없었기 때문에 fragment 까지만 처리가 되었다. IE5.5 이후에 모든 브라우저는 버퍼를 거치고 그림을 그리게 되었다. 버퍼는 2차원 배열이라고 생각하면 된다. 좌표의 색상이 무엇인지를 기억한다.

예전에는 그림을 CPU 가 그리면 그래프 파이프 라인을 타고 바로 그려졌다. 지금은 그림을 GPU 가 그린다 (대부분)

GPU 는 그림을 그릴 때, CPU 가 만들어낸 이미지를 GPU 에 올려줘야지만 그릴 수 있다. GPU 에게 그림을 주기 위해서라도 버퍼 구조를 만들게 된다. Post Process 단계를 CPU 에게 혹은 GPU 에게 넘겨줄 수 있다. GPU 가 훨씬더 빠르다.

모던 브라우저는 CPU 를 이용해서 Geometry fragment 를 한다. 그 이후에 가공한 버퍼를 GPU 에게 넘겨 준 후 PostProcess 를 처리한 후 디스플레이에 화면이 보여진다.

CPU 와 GPU 에게 일을 양분하면서 자원의 최대 효율을 끌어낼려고 한다.

post process 를 쓰는 대표적인 것은 ?

  • transform
  • keyframe animation..

이때 CPU 가 널널해진다.

Transform 3D#

Perspective (원근) 3D 를 나타내는 CSS 속성 중 하나이다.

모니터에서 사람눈까지의 거리를 원근의 거리라고 부른다. perspective:600px 은 잠정적으로 나의 눈과 모니터 사이의 거리를 600px 보겠다는 이야기 이다. 픽셀을 길게 줄수록 똑같은 물건이 작아보일 것이다.

단지 크기의 차이의 의미가 있는 것이 아니다. 어안렌즈 효과가 있다. 앞에 있는 것은 크게 보이는데, 조금만 옆에 있어도 대게작아보인다. 멀리있으면 왜곡이 덜한데 가까이 가면 왜곡이 심해진다. 멀리있으면 있을수록 마치 2D 처럼 flat 하게 보인다. 가까이 와야지만 3D 같아 보이고 왜곡효과가 커진다.

perspective Origin 이란?

변형을 할 때 어디를 바라보고 perspective 를 인식할 것이냐

perspective 는 거리만 나타낸다고 보면 이것은 모니터 한 가운데를 보고있다고 가정한다.

좌상단을 볼 때, 우 하단을 볼 때 이런 개념이 가능하다.

<html>
<body>
<div class="test"></div>
</body>
<style>
@keyframes origin {
0% { perspective-origin: 0% 50% }
50% { perspective-origin: 100% 50% }
100% { perspective-origin: 0% 50% }
}
@keyframes spin {
100% { transform: rotateX(360deg)}
}
body, html{ height: 100% }
body { perspective: 600px; background:#404040; animation: origin 10s linear infinite}
.test {
width: 500px; height: 500px; background: #aaa;
position: absolute; left: 50%; top:50%; margin-left:-250px; margin-top: -250px;
animation: spin 30s linear infinite;
}
</style>
</html>
<html>
<body>
<div class="test"></div>
</body>
<style>
@keyframes spin {
100% { transform: rotateX(360deg)}
}
body, html{ height: 100% }
body {
perspective: 600px; background:#404040;
}
.test {
width: 500px; height: 500px; background: #aaa;
position: absolute; left: 50%; top:50%; margin-left:-250px; margin-top: -250px;
animation: spin 30s linear infinite;
}
</style>
</html>
  • body 에 origin animation 만 넣은 상태는 static 하다.

세번째는 transform-style 이다.

이것은 DOM 의 tree 구조인 것이지 3D 의 트리구조가 아니다. fragment 단계 까지만 영향을 끼친다.

부모가 회전하면 자식도 회전하면 계층구조가 3D 에도 반영됬다고 할 수 있다.

preserve-3d 값을 쓰면, 부모의 3D 관련된 모든 속성을 이어 받을 수 있다.html

마지막으로 transform-origin 은 실제 변형이 일어날 때의 위치를 정한다.
