Graphics System#

그래픽스는 기본적으로 점을 하나 찍는 것부터 시작할 수 있다.

점을 왕창 찍으면

하얀색 만 찍으면 그림 같지 않아서 빨간 점을 찍었다.

각 점의 색깔 정보가 보인다.

각 점의 위치, 크기 정보도 보인다.

그래픽스 시스템의 근본적인 내용
→ X, Y, width, height, color

Fixed Number#

방금 전에는, 숫자가 확정 되어 있다 (고정된 숫자) 로 화면을 구성했다. 그래픽 시스템에서 Fixed Number 라고 한다.

고정된 숫자는 PC 태블릿 모바일에 적응되어 그릴 수가 없다.

고정된 숫자로 만들어진 그래픽스 시스템은 환경에 적응할 수 없다.

환경 : Screen Size, Chrome Size, Hierarchy 모니터 해상도, 브라우저의 창 사이즈, 스마트폰의 가로/세로 모드 등..

범용성 있는 그래픽스 시스템에 적합하지 않다.

Abstract Calculator#

이럴 경우 숫자를 대신할 매타포를 사용한다. : %, Left, Block, inline, float

퍼센트를 사용한다는 것은 비율로 계산한 건데, 부모.. 화면 을 기준으로 특정한 것을 기준으로 나타내는 것이다. 실제로 화면에 그려질 때 환경을 인식해서 숫자로 바뀌게 된다. % 는 공식이자 수식이다. 그림을 그리는 순간마다 함수로서 실제 숫자를 얻는다.

여기서 말하는 메타포를 한마디로 함수라고 이야기 할 수 있다.

Components#

하지만 단지 그것만으로 복잡한 그래픽을 그릴 수 없다. 그래서 abstract calculator 를 이어받는 것들을 만들어 낸다.

Framework#

Components 들이 일정한 규칙과 사용방법을 지키며 구현되어 있다면

Components 의 대표적인 예는 Html 이 있을 것이다. 그림을 그리기 위해 그림 이미지 경로의 픽셀을 얻어와서 만들지 않는다. 이미지 태그에 이미지를 넣는다.

html 태그 하나하나를 컴포넌트라고 부른다면 그런데 html 태그들은 스타일에 공통적인 속성을 먹는다. html 체계 전체를 프레임 워크라고 부를 수 있다.

부트스트랩의 하나하나를 컴포넌트라고 부른다면 부트스르랩 자체는 프레임웍이라고 볼 수있을 것이다.

그래픽스 시스템은 한마디로 점찍는 방법이라고 생각하면 된다.

Rendering System#

렌더링이라는 말은 그래픽스에서만 쓰는 것이 아니다. 데이터베이스의 내용을 Json 으로 렌더링.. 어떠한 대상이 있으면 내가 원하는 모습으로 다시 그려내는 것이다. 일반적으로 보다 구체적이고 시각적인형태로 바꿔주는 것.

여기에서 그림을 그리기 위한 데이터를 그림으로 바꿔내는 것을 의미한다 그림을 표현하기 위한 정보는 대체 무엇일까?

브라우저, 자바스크립트의 도움없이 만들 수 있을까?

보통 2단계를 통한다

Geometry Calculate#

1단계는 영역을 나누고 박스를 찾는 것이다.

Fragment Fill#

그러고 나서 색칠을 한다.

색칠하는 과정에서 하나하나 칠하는 그 대상을 Fragment 라고 한다.

픽셀이라는 단어는 중립적이지 않아서 쓰지 않음 일반적으로 핸드폰에서 HTML 뷰포트를 scale 로 했을 때 나오는 해상도가 과연 핸드폰의 픽셀 해상도와 일치할까? 아니다 훨씬 더 많다. 그때의 1px 은 무엇일까? QHD 에서 실제 물리적인 화소 4개가 1픽셀이된다. 레티나의 원리가 폰트 12px 을 영어로 썼는데 이쁘게 나온다. 12공간안에 점이 엄청 많을 것이다. 이때의 픽셀은 무엇인가? 픽셀의 비율이라고 부른다.

1픽셀을 채우는 것이 아니라 해당 물리적인 점을 다 채워야 한다. 모든 공간을 가리키는 중립적인 단어를 frament fragment 를 채운다.

렌더링은, geometry 계산하고 fragment 를 채운다.

브라우저에서 geometry 계산 하는 과정을 reflow 라고 부른다. reflow 된 영역안에 채우는 행위를 repaint 라고 부른다.

CSS Specifications#

렌더링 시스템과 그래픽 시스템을 이야기하는 이유는 ? CSS 는 어떻게 하면 고정숫자를 사용하지 않고 계산된 체계로 그래픽을 표현할까와 지오매트릭 영역을 어떻게 표현하고 색칠을 할때 어떤 식으로 할까에 대한 일종의 언어 인 것이다.

CSS 를 배운다는 것을 다시 말하자면, cSS 속성 값이 구체적으로 발현될때는 어떤방식으로 계산되서 표현되는지 이해하는 것이다.

CSS 는 다른 시스템에서 방법을 차용을 많이 하였다. 따라서 복잡한 체계를 가지게 되었다.

표준을 이해해야 하는것이 중요하다.

CSS level1#

오페라 브라우저가 W3C 에 제안했다. HTML 에 그림관련된 것을 집어넣었더니 복잡해졌다. 기존 그래픽스 시스템을 차용해서 CSS 시스템을 만들자고 제안.

CSS level2 + module#

MS IE4 가 많이 제안되었다.

CSS 는 그림에 대한 것인데 이것을 하나의 스펙으로 관리하는 것은 무리라고 판단. 그래서 CSS2 스펙 안에, 관심 분야별로 모듈 시스템을 넣어서 사양을 모듈 별로 관리 함

CSS 는 다양한 기능을 갖고 있다. 윈도우 렌더링 시스템 X윈도우,, 맥.. 렌더링 시스템.. 등 분야가 많으니깐 단일 사양이 어렵기 때문에

모듈로 관리하였다.

CSS level2.1#

CSS 의 모든 모듈들이 2로 고정이 되었다.

모듈별로 어던 분과는 열심히 해서 모듈 3 를 발표하는 팀이 생겨버렸다. 반대로 변화가 거의 없는 팀도 존재했다.

따라서 CSS2.1 이후에 CSS3 로 가는 것이 불가능 하다는 것을 깨달았다.

벌써 level3 가 된 모듈들은 Syntax, cascade, color, selectors, background values, text, text decor, fonts, ui

흔히 css3 라고 부르는 것은 CSS2.1 에 level3 이 된 모듈들을 모아서 css3 이라고 부를 뿐이다. 공식문서에 css3 는 존재하지 않는다.

Module Level#

더이상 CSS ..level 이 존재하지 않는다.

새로 태어난 모듈들은 이때 transforms1 compositing1 effects 1 masking 1 flexbox1 grid 1 1레벨을 갖고 있다.

브라우저마다 사용할 수 있는 모듈의 레벨도 다르다.

Other Specifications#

W3C 가 영향력을 갖고 있던 시대는 아이폰을 만들기 전이다. MS 의 영향력이 약화되고 구글하고 애플의 영향이 강화됐다.

새로운 신흥 강자들이 반발을 제기했다. HTML5 같은 경우 W3C 와 관계없이 브라우저에 탑재 되었다.

W3C 도 W3C Community and Business Groups 를 운영하게 됐다. 표준은 아니지만 W3C 산하의 커뮤니티로 인정하고 이 활동을 공식적인 활동으로 인정해준다.

이제 별개의 그룹보다 W3C 커뮤니티 산하의 그룹을 만들게 되었다. 이 그룹에서 사양을 만들에서 W3C 에다가 draft 로 밀어 낸다.

이 그룹들에서 만든 스펙을 삼성 브라우저, 사파리, 크롬.. 에다가 넣고 있다.

이중에서 CSS 에 관련이 깊고 영향력이 있는 그룹은 WICG Web platform incubator community group RICG 가 있다.Respossive issues community group

WICG 가 파워가 쎄서 RICG 는 거의 WICG 에 먹혀 버렸다. 파워가 쌘 이유는 구글이 주 멤버라서

WICG 에 나온 draft 는 크롬에 먼저 반영되고 W3C 에 반영될 정도

현재 스펙을 이해하려면 W3C 스펙만 보기에도 어렵다.

Normal flow#

CSS 에 공식적인 고유 명사이다.

CSS2.1 Visual Formatting Model Positioning Schemes & Normal Flow

Position#

static / relative / absolute / fixed / inherit

그래픽스 시스템에서 추상적인 계산 책에 배웠다

포지션은 지오매트리 영역에 left top 를 계산할 때 추상적인 위치를 결정하게 하는 시스템이다.

normal flow 는 position 이 static relative 일 때만 가능하다.

normal flow 또한 두가지로 설명이 된다. normal flow 에는 두가지 계산공식이 있다.

첫번째는 Block Formatting Contexts (BFC)

두번째는 Inline Formatting Contexts (IFC)

세번째는 Relative Positioning -> normal flow 의 일부이지만 포지셔닝을 정의하고 있다.

실제 nomalflow 를 계산하는 방법은 Block ... Inline 이라고 할 수 있다.

block 과 lnline 은 ?

block 은 부모의 가로길이를 가득채운 한 줄이다. BFC 는 한줄을 다먹을때 어떤식으로 계산하는 가 x 는 언제나 0 이고 width 는 언제나 부모의 width 이다.

그러면 bfc 에고 유일하게 고민해야 할 것은 .

다음번 블록의 y 가 어디냐 이다. 첫번째 블록 context 의 height 이다 .

세번째 는 첫번째 두번째 height 를 합친 값이다. 이 계산 방법을 bfc 라고 한다.

bfc 의 두번째 특성은, 안에 있는 블록요소의 height 의 합이 나의 height 이다.

ifc 은 한라인을 먹지 않고 나의 컨텐츠 크기만큼 width 를 차지한다.

첫번째 inline 의 다음 두번째 inline 은 첫번째의 width 만큼 떨어진 곳이 x 가 된다.

ifc 공식 중 하나는 인라인 요소의 width 의 합이 부모의 width 를 넘어가면 다음줄로 넘어간다.
얼마큼 내려와야 할 까? 지금 인라인을 구성하는 것중 가장 큰 height 가 line height 가 되어서 line height 만큼 y 값이 내려오는 공식

그러나 ifc 영역이 생겨나도 block 이 오면 ifc 영역이 종료되고 다음번 bfc 블록이 생긴다.

<div style="width:500px;backgroud:red">&nbsp;</div>
<div style="background:blue">&nbsp;</div>

aaaaaa... 는 ifc 인데 아래로 넘어가지 않고 부모의 width 를 넘어가면 안되는데, 왜 아래로 안내려가고 부모의 width 를 넘어갈까?

<div style="width:200px; background: red">
aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa
</div>

암묵적으로 공백문자가 없는 문자를 하나의 ifc section 으로 본다.

<div style="width:200px">
aaaaaaaaaa
aaaaaaaaaa
aaaaa
</div>

이라면 공백으로 구분된 각각의 요소를 inline 하나의 요소로 본다.

만약에 공백문자를 사용하고 싶지 않으면 div 에 특별한 속성을 지정해야 한다.(word break) 이 공간 안에서는 공백문자 없이 붙어있는 문자열도 하나하나 문자를 인라인으로 처리해라 라는 것. 당연히 브라우저가 엄청 느려진다.

왜 느려질까? geometry 영역이 많아짐으로써 계산 영역이 많아진다.

BFC + IFC

인라인 안의 블록 요소 가 있다. 렌더링 시스템과 돔의 구조는 무관하다. 돔의 포함관계와 렌더링을 평가할때 포함관계는 다르다. 렌더링은 DOM 의 요소 구조를 보지 않고 BFC IFC 구조만 본다.

span 태그에 relative position 을 입히면 relative position model 로 바뀐다.

static 에서 상대적으로 위치를 이동시킨다.

position static 과 relative 가 섞여 있으면, 무조건 position relative 가 static 위로 올라온다.

relative 의 geometry 계산은 static 가 동일하다.

relative 은 static 을 그린다음에 상대적으로 옮겨준 것이다.

normal flow 이름은 브라우저에서 그림그릴 때 가장많이 쓰는 공식이기 때문이다.
