프로그램을 짰을 떄 어떤 부분을 해결하기 위한 핵심적인 지식 부분이다. 그 문제가 보다 추상적인 개념에서 범용으로 적용할 수 있다고하면 지식부분만 바꿔주면 얼마든지 처리할 수 있다.

지난 시간의 바인더 전체 코드이다. 이 코드는 하나로 보이겠지만 사실은 그렇지 않다.

바인더의 필드와, 필드의 구조,ㄱ 구조를 루프돌고 있는 부분 이것을 Structure 혹은 Structure control 이라고 한ㄷ. 객체는 이러한 자료구조를 가지고 있기때문에, 자료구조를 순회하기 위해서는 이러한 구조가 될 수 밖에 없다. 바인더가 어떻게 짜여져 있느냐의 문제이다. 바인더 내부에 데이터 구조가 어떻게 되어있느냐

객체는 상태측면과 행동측면이 있다. 필드값들은 자료를 받았을 때 자료를 순회..조회 하기 위해 등. 자료의 모양 때문에 생겨나는 구조가 있다. item 은 Set 을 갖고 있기 때문에 forEach 를 돈 것이다. 뷰모델은 안에 스타일이나 ,.. 프로퍼티 등등이 있다..

여기까지 문제해결전략이 아닌 구조의 문제다.

그에 비해서, 이부분은 전략에 관련된 문제다. 특정도메인을 해결하기위한 알고리즘 지식혹은 도메인이 전략이다. 전략 부분을 외재화 할 것이다. 하지만 전략부분의 일부는 구조의 문제에 항상 관여한다. 절대 분리되어 전략이 작동될 수 없다.

객체지향에서 객체는 자신의 상태를 갖고 자신만의 행위를 하기 떄문에 상태와 행위를 분리할 수 없다. 언제나 상태의 자료구조를 갖고 있을 떄만 전략이 성립한다.

특정한 자료구조가 특정한 알고리즘과 매핑된다는 것이다. 이 전략은 처음부터 특정한 데이터를 지칭할 수 있는 포인터를 갖고있어야 매핑할 수 있는 알고리즘을 만들 수 있다.

이것은 흔히 짜고 있는 코드로부터 전략을 빼내는 발상법의 아이디어가 된다. 객체지향에서 이 부분을 컴포지션이라는 것으로 해결을 하는데 이것은, 코드를 객체로 바꾸는 것이다. 코드를 바꾸면 회귀 테스트를 해야한다. 하지만 코드를 안바꾸고 객체로 바꾸고자 한다. 전략을 먼저 도출하는데 성공했으면 전략이 구조와의 관계를 생각하고 코드를 객체로 바꾼다. 객체를 도출할때 인터페이스와 클래스로 도출해야 한다. 형에 대한 규격에 따라 코드가 반응할 수 있게

이 때 코드가 객체가 어떤 식인지 알아야 쓸 수 있다. 이것을 타입이라고 한다. 타입을 정의해서 객체를 받아 들이면 기존 코드에는 의존성이 없었는데 객체가 알아야지만 렌더가 작동할 수 있게 된다. 코드 부분은 외부 객체의 컴포지션으로 바꿀 것이다. 그러면 이 객체에 대한 지식이 있어야지만 바인더가 작동할 수 있기 때문에 바인더는 전략객체에 대한 의존성이 생긴다. 객체지향에서 의존성 발생이유는, 위임했기 때문이 아닌 나의 전략을 외부 객체의 도움으로 해결하고 싶고 그렇게 하지 않으면 변화가 생길 때 마다 코드를 바꿔줘야 하기 때문에 코드부분을 객체로 바꿔주면 객체지향은 자연스럽게 객체 의존성이 생긴다.

바인더는 바인더 전략과 의존성이 생길 수 밖에 없다. 의존성이 생겼을 때 내부에서 의존성을 외부에서 공급받는 것이 좋다. DI 외부에서 만들 경우 특정 개체에 대한 의존성이 생긴다. 타입에 대한 의존성만 있는 것이 낫다. 결국 코드도 고쳐야 한다.

결국 코드를 밖으로 꺼내는 순간 DI D 가 자동으로 생긴다.

일반성을 도출해야 한다. 그리고 전략을 상속 받은 객체에서 위임하려고 한다. (override) 이것을 탬플릿 메서드 라고 한다. 외부에는 process 가 템플릿 메서드 (외부에서는 process 로 보인다) 내부에서는 자식쪽으로 injection 된 메서드에 의존한다. _process 자식쪽에 의존하고 있는 메서드는 템플릿 메서드에서 훅이라고 한다.

객체지향은 문제를 객체망이 해결한다. 객체망을 구성하는 것은 어떤 객체가 다른 객체를 필드수준으로 알고있거나 메서드 수준에서 알고있거나 자식으로 알고있거나 3개밖에 없다.

프로세서를 알고있는 자식 4개를 만들었다.

숙련이 필요하다.

다른 객체지만 4가지 구상 인스턴스들이 제공해 주는 것은, 4가지를 프로세서라는 형으로 인식할 수 있게 해준다.

프로세서가 제공해 주는 것은

이제 바인더를 개조할 것임.

#processors 를 cat 으로 잡지 않았다. v.cat 이라는 카테고리 당 하나의 프로세서만 장착시킬려고. 스타일스라는 캣을 갖고 있는 스타일을 처리해주는 프로세서를 만들었다. 비슷한 키가 똑같은 스타일스라는 키를 갖고잇는 것이 또 들어오면? 셋 안에 객체로서는 다르지만 캣이라는 프로세서가 같은 것이 들어온다. 스타일스라는 키를 누구한테 위임해야 할지 헷갈리는 것을 방지하기 위해 오브젝트를 썼다. 자료구조를 신중하게 선택해야 한다.

객체는 원래 객체로 식별 되어야 하는데, 이번에는 값을 사용한 것이다. 왜 그럴 까? 값을 안쓸수도 있다. v.cat 이 스트링이 아닌 미리 정의한 심볼을 받으면 된다. 값을 방어하기 위한 다른 체계를 만들어 줘야 한다. 값을 쓰면 계속 힘들어진다. 그럼에도 불구하고 뷰모델이 무한의 키를 가질 수 있다. 무한의 키가 자바스크립트 언어수준으로 객체니까 키가 문자열로 되어 있어서 어쩔수없이 문자열로 맞췄다. 다음시간에서는 심볼로 전환할 것임. 프로세서는 객체지만 외부에 노출된 v.cat 은 값이기 때문 값을 외부에 노출할 때는 신중하게 생각해야 한다. 프로세서를 등록할 수 있게 되었다.

자바스크립트는 싱글 스레드 머신이기 때문에 메서드가 도는 동안에 중간에 addProcessor 가 일어날 확률은 없다. 그래서 const processores = Object.entries(this.#processors); 만 한번 조사하면 루프를 처리하는 동안 processores 가 늘어나거나 줄어들 경우가 없다. 키 값 이차원 배열을 processores 에 잡아논것임. processores 를 아이템 마다 무엇을 하는가 보면 아이템 안에 프로세서스를 키 값으로 잡아놨는데 pk 에 cat 이 들어있고 processor는 v 값이 들어있을 것이다. 해당 캣에 들어있는 프로세서가 들어있을 것인데, vm[pk] vm 에 있는 캣을 찾아서 이것이 바로 대상 오브젝트의 구조가 되는 것이다. 이것의 forEach 를 돌면 vm 의 키 갑이 나올 것이고 k v 를 프로세서한테 넘겨주면 처리가 된다.

const Binder = class extends ViewModelListener {
#items = new Set;
#processors = {};
add(v, _ = type(v, BinderItem)) {
this.#items.add(v);
}
addProcessor(v, _0 = type(v, Processor)) {
this.#processors[v.cat] = v;
}
render(viewmodel, _ = type(viewmodel, ViewModel)) {
const processores = Object.entries(this.#processors);
this.#items.forEach(item => {
const vm = type(viewmodel[item.viewmodel], ViewModel), el = item.el;
processores.forEach(([pk, processor]) => {
Object.entries(vm[pk]).forEach(([k, v]) => {
processor.process(vm, el, k, v);
});
});
});
}
}

그럼 내가 사용하고 있는 프로세서의 형에 대한 지식을 봅시다. 프로세서인 것을 알 고 있으니까 cat 을 사용하였고 프로세서인 것을 알고 있으니까 프로세스라는 메서드를 사용한 것이다. Processor 에서 정의하고 있는 프로토콜만 사용해서 알고리즘을 바꿨다. 이것이 바로 객체 사이의 의존성 또는 계약이라고 부르는 것이다.

바인더는 프로세서랑 계약을 했기 때문에 프로세서 명세서를 ㄱ쓸 수 있는 것임 바인더는 내부의 코드를 프로세서와의 계약으로 풀었기 때문에 알고리즘을 프로세서에서 공개하고있는 계약 내용만 갖는 알고리즘으로 바꿔줘야 한다. -> 알고리즘의 일반화(추상화) 나의 복잡한 코드들의 공통점을 모아서 객체로 뺀 뒤에 객체를 형으로 뺀 뒤에 형과 계약하고 형으로 계약한내용으로 알고리즘을 바꾸는 것을 의미한다. -> 제네릭 알고리즘

전략부분을 형으로 빼낸다. 형과 계약한 내용으로 형이 제공하고 있는 메서드로만 이용해서 알고리즘을 고친것. 이 코드는 프로세서의 메서드나 속성이 바뀌지 않는 이상, 바인더는 더이상 코드를 고칠필요가없다. 대신 수 많은 전략들을 추가하거나 바꿀 수 있다. 이것이 전략 패턴이다. 이 프로토콜의 내용이 적당하고 적어야 한다.

  1. 구조적인 부분과 전략적인 부분을 나눈다.
  2. 전략부분의 공통부분을 찾느다.
  3. 전략이 어떻게 상태와 관계를 맺는지 찾는다
  4. 이것들로 도출된 형으로 가지고 알고리즘을 고친다.

바인더가 프로세서를 의존하게 되었다. 객체제엇 의존서을 선택할 때, 단방향 의존성을 설정해야 한다. 객체지향에서 의존성이 없다는 얘기는 코드를 발랐다? 라는 것이고, 코드를 맨날 고쳐야 된다는 이야기다. 객체지향을 쓰는 이유는 코드 부분을 객체로 바꿔서 객체만따로 공급하면 그 코드를 건들 필요가 없게 하기 위해서 이다. 의존성이 많은 코드가 나쁜 게 아니라. 의존성이 한 방향으로 흐르지 않고 양방형으로 흐르는 것이 나쁜 것이다. 의존성은 당연히 생기는 것이다. 제대로 짰으면, DI 가 자동으로 따라온다.

코드를 객체로 바꾸면 의존성이 따라오고, 의존성이 생겼는데 DI 메서드가 없다면 잘못된 것이다.

Observer#

지금까지는 이렇게 코드를 짰다 직접 call

하지만 많은 프레임 워크에서는 자동으로 해준다. 바인더가 뷰 모델을 옵저빙 하기 때문이다. 바인더는 뷰모델의 변화를 감지하고 있다가 변하면 바인더를 랜더링함

하지만 실제로, 이렇다. 옵저버 패턴에서 조심해야 할 것은. 옵저버가 일을 열심히 하는 것 같지만, 감시하는 사람은 안피곤하고 뷰 모델이 더 피곤하다. 실제로는 뷰 모델이 바인더에게 notify 한다. 뷰 모델의 변화가 일어나면 분류해서 보내줘야 한다. 더 큰 문제는 뷰모델의 변화를 스스로 알수 있느냐다. 내 변화를 알아내는 것이 힘들다.

바인더.렌더가 없어졌다. 뷰모델의 값만 바꾼다. 완전한 버전의 MVVM 이다.. MVVM 은 뷰모델만 건드렸다....

바인더를 더이상 call 하지않고 notification 을 한프레임에 한번하게 된다.

현실세계에서 쓰기 힘들다. 전략패턴 탬플릿메서드패턴 옵저브 컴포짓 패턴만 잘써도 많은 문제를 해결할 수 있다.

다음시간에는 배열받아서 돔엘리먼트를 만들어주는것을 할 것이다.
